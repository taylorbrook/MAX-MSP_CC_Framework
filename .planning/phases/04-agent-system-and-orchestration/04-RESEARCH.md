## Summary

Phase 4 builds the orchestration layer that transforms the existing patch/code generation library (Phases 1-3) into a usable multi-project development environment. The core technical challenge is not complex algorithms but rather correct architecture: defining agent prompts that load the right context slices, critic loops that extend (not replace) the existing validation pipeline, memory that persists across sessions, and slash commands that wire it all together.

The implementation model is well-proven. The user's Plugin Freedom System (PFS) at `/Users/taylorbrook/Dev/VST-development/` provides a battle-tested reference architecture with 26+ skills and 44+ commands. Claude Code's native skill/command system uses `.claude/commands/` for slash command definitions (markdown files with frontmatter) and `.claude/skills/` for skill definitions (SKILL.md with references/). This is the exact infrastructure to use -- no custom framework needed.

**Primary recommendation:** Implement agents as Claude Code skills under `.claude/skills/`, slash commands as `.claude/commands/` markdown files, memory as markdown files in `~/.claude/max-memory/` (global) and `patches/{project}/.max-memory/` (project), and critics as Python modules in `src/maxpat/critics/` that extend the existing validation pipeline.

### Pattern 1: Slash Command -> Skill Dispatch
**What:** Each `/max:*` slash command is a markdown file in `.claude/commands/` that routes to the appropriate skill.
**When to use:** Every user-facing command.
**Example:**
```markdown
# .claude/commands/max-build.md
---
name: max-build
description: Generate patch/code via agent dispatch
argument-hint: "[description of what to build]"
---

# /max:build

Generate a MAX patch or code based on the user's description.

## Behavior

1. **Load active project** from `patches/.active-project.json`
2. **If no active project:** Prompt user to run `/max:new` first
3. **Invoke max-router skill** with:
   - User's build description
   - Active project context (context.md, .max-memory/)
   - Global memory (filtered by domain relevance)
4. **Router dispatches** to specialist agent(s)
5. **Run critic loop** on generated output
6. **Write output** to project's generated/ directory
7. **Write-back memory** if generation was successful

## Arguments
- `description` - What to build (required, natural language)

## Examples
/max:build "subtractive synth with filter envelope"
/max:build "step sequencer that outputs MIDI"
/max:build "gen~ waveshaper with drive and mix controls"
```

### Anti-Patterns to Avoid
- **Monolithic agent prompt:** Don't put all 6 agents in a single SKILL.md. Each agent is its own skill directory with focused context loading.
- **Loading all object JSON in every agent:** Don't load all 2,012 objects for the js/Node agent. Each agent loads only its relevant domain slices (e.g., DSP agent loads msp/ and gen/ only).
- **Critic as LLM-only judgment:** Don't rely solely on LLM reasoning for critic checks. Put deterministic checks (inlet/outlet counting, signal flow tracing) in Python, leave semantic judgment (architecture quality, pattern recognition) to the critic skill prompt.
- **Memory as unstructured dump:** Don't append raw text to memory files. Use structured sections with dates, contexts, and actionable rules.
- **Re-implementing validation in critics:** Critics EXTEND the pipeline, they don't REPLACE it. Never duplicate checks already in validation.py.

## Common Pitfalls

### Pitfall 1: Context Window Bloat
**What goes wrong:** Loading all agent context (all object DBs, all memory, full CLAUDE.md) for every operation exhausts the context window, degrading quality.
**Why it happens:** Eager loading of "everything that might be relevant" instead of targeted loading.
**How to avoid:** Each agent skill specifies exactly which files to load. DSP agent loads msp/ and gen/ objects (437 objects) not all 2,012. Router loads only dispatch rules, not generation context. Memory injection filters by domain tag.
**Warning signs:** Agent responses become vague or miss specifics. Context window warnings appear.

### Pitfall 2: Critic Loop Infinite Cycling
**What goes wrong:** Decision specifies "no round limit -- critics keep requesting revisions until all issues resolved." This can loop forever if critic and generator disagree.
**Why it happens:** Critic finds issue X, generator fixes X but introduces Y, critic finds Y, generator fixes Y but reintroduces X.
**How to avoid:** Track revision history. If the same finding appears twice after being "fixed," escalate to user with both positions. Practical safety: 5 revision rounds max before escalation, even though the decision says "no round limit." The spirit is "don't artificially cut off" -- but infinite loops are a real risk.
**Warning signs:** Same critic finding appearing in consecutive rounds. Revision count exceeding 3.

### Pitfall 3: Memory Deduplication False Negatives
**What goes wrong:** Similar patterns stored multiple times because exact text matching misses semantic duplicates ("user likes line~ for gain" vs "prefer line~ for smooth gain transitions").
**Why it happens:** String comparison on markdown entries misses meaning equivalence.
**How to avoid:** Dedup by domain + behavior hash, not exact text. When writing, present the candidate entry alongside existing entries in the same domain and let the agent decide if it is genuinely new. Favor fewer, higher-quality entries.
**Warning signs:** Memory files growing beyond ~50 entries per domain. Contradictory patterns stored.

### Pitfall 4: Agent Boundary Confusion
**What goes wrong:** Patch agent generates gen~ code, DSP agent generates patch structure, outputs conflict.
**Why it happens:** Unclear boundaries between agents for tasks that span domains.
**How to avoid:** BOUNDARIES.md for every agent explicitly lists what it does and does NOT do. Router's merge protocol handles cross-domain coordination. For multi-agent tasks, one agent is designated "lead" (usually the one whose domain dominates).
**Warning signs:** Two agents producing the same file. Conflicting inlet/outlet counts between gen~ code and its wrapper.

### Pitfall 5: RNBO/External Agent Stubs Breaking
**What goes wrong:** RNBO~ and Externals agents are stubs in Phase 4 (real implementation in Phase 5), but users invoke /max:build with RNBO tasks.
**Why it happens:** Router dispatches to stub agent, which has no generation capability.
**How to avoid:** Stub agents return a clear message: "RNBO~ generation is planned for Phase 5. For now, I can help you design the patch structure, but export-specific features are not yet implemented." Do not silently fail.
**Warning signs:** Empty or error outputs from RNBO/external tasks.

### Pitfall 6: Project State Desync
**What goes wrong:** Active project tracking file says "my-synth" but user is working in a different project directory.
**Why it happens:** User manually creates/moves files without using /max:switch, or crashes during project operations.
**How to avoid:** Every command reads `.active-project.json` and validates the project directory exists. If desync detected, prompt user to resolve. /max:status always shows current active project prominently.
**Warning signs:** Commands writing to wrong project directory. Status showing stale data.

### Example 2: DSP Critic - gen~ I/O Matching
```python
# src/maxpat/critics/dsp_critic.py (partial)

def _check_gen_io_match(patch_dict: dict, code_context: dict | None) -> list[CriticResult]:
    """Check that gen~ codebox I/O matches wrapper box I/O.

    A common error: GenExpr code uses in1, in2, in3 (3 inputs) but the
    gen~ box in the patch only has 2 inlets.
    """
    results = []
    if not code_context or "genexpr" not in code_context:
        return results

    from src.maxpat.codegen import parse_genexpr_io
    code_inputs, code_outputs = parse_genexpr_io(code_context["genexpr"])

    # Find gen~ boxes in the patch
    for box_entry in patch_dict.get("patcher", {}).get("boxes", []):
        box = box_entry.get("box", {})
        if box.get("maxclass") == "gen~":
            patch_inputs = box.get("numinlets", 0)
            patch_outputs = box.get("numoutlets", 0)

            if code_inputs != patch_inputs:
                results.append(CriticResult(
                    "blocker",
                    f"gen~ I/O mismatch: GenExpr code uses {code_inputs} inputs "
                    f"but gen~ box has {patch_inputs} inlets",
                    f"Update gen~ box to have {code_inputs} inlets, or "
                    f"modify GenExpr to use {patch_inputs} inputs",
                ))

            if code_outputs != patch_outputs:
                results.append(CriticResult(
                    "blocker",
                    f"gen~ I/O mismatch: GenExpr code uses {code_outputs} outputs "
                    f"but gen~ box has {patch_outputs} outlets",
                    f"Update gen~ box to have {code_outputs} outlets, or "
                    f"modify GenExpr to use {patch_outputs} outputs",
                ))

    return results
```

### Example 3: Structure Critic - Missing Trigger Objects
```python
# src/maxpat/critics/structure_critic.py (partial)

def _check_fan_out_without_trigger(patch_dict: dict) -> list[CriticResult]:
    """Detect outlets connected to multiple inlets without trigger object.

    Per CLAUDE.md Rule #4: Use explicit trigger objects for fan-out instead
    of connecting one outlet to multiple inlets.
    """
    results = []
    lines = patch_dict.get("patcher", {}).get("lines", [])

    # Count connections per source (source_id, outlet_index)
    source_counts: dict[tuple[str, int], int] = {}
    for line_entry in lines:
        pl = line_entry.get("patchline", {})
        source = pl.get("source", [])
        if len(source) >= 2:
            key = (source[0], source[1])
            source_counts[key] = source_counts.get(key, 0) + 1

    # Find fan-outs (same outlet -> 2+ destinations)
    for (src_id, outlet_idx), count in source_counts.items():
        if count >= 2:
            # Check if source is already a trigger object
            src_name = _get_box_name_by_id(patch_dict, src_id)
            if src_name in ("trigger", "t"):
                continue  # trigger objects are the solution, not the problem

            results.append(CriticResult(
                "warning",
                f"Fan-out without trigger: '{src_name}' ({src_id}) outlet "
                f"{outlet_idx} connects to {count} destinations",
                f"Consider using a 'trigger' (t) object to make the execution "
                f"order explicit. Current connections may execute in unpredictable order.",
            ))

    return results
```

## Open Questions

1. **Task tool availability for agent delegation**
   - What we know: PFS uses `Task` tool to invoke subagents. This is a Claude Code feature for spawning sub-conversations.
   - What's unclear: Whether Task tool is available in this project's Claude Code environment (it may be an experimental feature).
   - Recommendation: Design for Task-based delegation but include a fallback where the router skill directly loads and executes the specialist skill's instructions inline (no subagent spawn). The router skill would become more monolithic but still functional.

2. **Slash command naming convention**
   - What we know: Commands go in `.claude/commands/` as markdown files. PFS uses `/implement`, `/start`, `/test` etc.
   - What's unclear: Whether colon syntax `/max:build` works in command file naming or if it must be `max-build`.
   - Recommendation: Use `max-build.md` as filename. The command name in frontmatter can be `max-build` or `max:build` -- test both. PFS uses hyphenated names.

3. **Critic loop execution model**
   - What we know: Generate -> critic review -> revise loop runs until clean.
   - What's unclear: Whether critics should run as separate Task invocations or as inline Python calls within the generator agent's context.
   - Recommendation: Python critic modules run inline (fast, deterministic). The critic skill orchestrator runs the loop logic. This keeps the loop in a single context window, avoiding spawn overhead.

<decisions>
## Implementation Decisions

### Agent specialization & dispatch
- Auto-dispatch: user describes what they want, a router agent analyzes the task and dispatches to the right domain agent(s) automatically
- Router uses task analysis (keyword/intent matching against domain definitions) to decide which agents to invoke
- 6 specialist agents: Patch (Max objects/routing/subpatchers), DSP/Gen~ (GenExpr/signal chains), RNBO~ (export-aware patches), js/Node (JavaScript for MAX), Externals (C/C++ SDK), UI/Layout (presentation mode/controls)
- Each agent loads domain-relevant context: ObjectDatabase slices, relevant Python modules, domain-specific object JSON
- Multi-domain tasks dispatched to multiple agents in parallel, then a merge step combines outputs and resolves conflicts (e.g., inlet counts must match between gen~ code and patch wrapper)

### Critic loop behavior
- Post-generation review: generator produces output first, then critic agents review it. Critics can request revisions, creating a generate → review → revise loop
- Loop runs until clean — no round limit. Critics keep requesting revisions until all issues resolved
- Two separate critics: DSP critic (signal flow, audio rate consistency, feedback loop safety, gain staging, gen~ I/O matching) and Structure critic (object usage/args, subpatcher organization, redundant connections, hot/cold inlet ordering, missing trigger objects)
- Inline annotations for any unresolved issues: comment objects in .maxpat, code comments in GenExpr/js, plus console summary
- Critics extend (not replace) the existing Phase 2 validation pipeline — validation catches mechanical issues, critics catch semantic/architectural issues

### Memory scope & persistence
- Dual scope: project-specific memory (patterns for this MAX project) and global memory (cross-project user conventions)
- Global memory at ~/.claude/max-memory/, project memory at project/.max-memory/
- Stored as readable markdown files
- Auto write-back at end of each successful generation — agents review what was built and write notable patterns
- Deduplication on write — check existing entries before adding new ones
- Auto-inject relevant memory at task start: project memory always loaded, global memory filtered by domain relevance
- Viewable and editable via /max:memory command — list, view, forget entries

### Project lifecycle & commands
- Full lifecycle mirroring GSD's discuss → research → plan → build → verify flow:
  - `/max:new "name"` — Create project with conversational kickoff (describe what you want, Claude asks clarifying questions, creates project directory)
  - `/max:discuss` — Capture implementation decisions (like discuss-phase)
  - `/max:research` — Investigate MAX-specific approaches (object choices, signal flow patterns, existing solutions)
  - `/max:build "desc"` — Generate patch/code via agent dispatch
  - `/max:iterate` — Modify existing patches/code
  - `/max:verify` — Run critic loop on existing output
  - `/max:test` — Generate step-by-step manual test checklist for MAX validation
  - `/max:status` — Project overview and progress
  - `/max:memory` — View/edit stored memories
  - `/max:switch "name"` — Change active project
- Directory-based multi-project isolation: each project gets its own directory under patches/ with context.md, .max-memory/, status.md, and all generated files
- /max:switch changes the active project — all subsequent commands target that project's directory

### Manual testing protocol
- Step-by-step numbered checklist generated by /max:test
- Each step: action to take in MAX + expected result
- User reports pass/fail per step
- Results stored in project status

### Claude's Discretion
- Agent prompt engineering and tool access configuration
- Router classification implementation details
- Memory storage format and deduplication algorithm
- Merge step conflict resolution strategy for parallel agents
- Critic severity levels and annotation formatting
- Project directory structure internals beyond the documented layout

</decisions>
